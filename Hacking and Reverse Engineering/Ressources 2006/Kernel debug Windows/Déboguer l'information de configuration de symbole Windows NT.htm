Informations générales

Les fichiers de symboles de débogage (symboles) sont nécessaires pour effectuer le noyau et l'utilisateur le débogage dans Symboles Windows NT. constitue un moyen de référencer des variables globales et de fonctionner des noms dans l'exécutable chargé.

Des symboles sont provoqués par l'éditeur de liens. Ils sont supprimés de produit commercial et sont enregistrés dans un fichier (.DBG) distinct. Cette opération permet de réduire considérablement la taille des fichiers et donc de diminuer le temps de chargement des fichiers et d'améliorer les performances du système Il réduit également le nombre de disquettes d'installation. Symboles représentent les noms Function/API et les variables globales.

Le .DBG File contient l'information symbolique pour chaque fichier. [ Les se trouvent sur le CD-ROM d'installation dans \Support\Debug\[i386| mips ] \Symbols. Ils figurent également sur le serveur NT de génération. L'emplacement est \\Ntbuilds\Release\Usa\Build###\[x86| |alpha]\Fre.srv\symbols mips.

Le répertoire Symbol est répartir dans sept sous-répertoire, appelés Extension Subdirectories (remarque que plusieurs les fichiers de ces répertoires de symboles sont mode Utilisateur) :

COM - répertoire pour les symboles de tous les fichiers se terminant par .CO
CPL \ash répertoire pour les symboles de tous les fichiers se terminant par .CPL
DLL - répertoire pour les symboles de tous les fichiers se terminant par .DL
DRV - répertoire pour les symboles de tous les fichiers se terminant par .DR
EXE - répertoire pour les symboles de tous les fichiers se terminant par .EX
SCR symboles de tous les fichiers se terminant par .SCR apparaissent ici
SYS - répertoire pour les symboles de tous les fichiers se terminant par .SYS

Symboles doivent correspondre à des versions de fichier :

Symboles d'une version différente fournissent l'information erronée, envoient à développeur de chasser des ombres et perdent l'heure considérable. Double check avec le client que que crée le client exécute et que le client possède tous les correctifs a installé. Le Kernel Stop Screen affiche le numéro de version du noyau.

Générations corrigées :

Les versions avec correctifs telles que les Service Packs requièrent un jeu de symboles spécial associant les symboles de la version de base et ceux des correctifs

Configurer les arborescence de symboles personnalisées

N'oubliez pas que symboles doivent correspondre aux fichiers installés sur l'ordinateur d'un client. Fréquence vous devez créer des symboles ensemble personnalisé simplement pour un client particulier à laquelle.

REMARQUE : Symbol Trees complet peut durer de 30 Mo d'espace disque plus de.
1. Créez un sous-répertoire pour stocker votre jeu client de symbole. Par exemple

C:\MYSYMBOLS
2. Démarrez toujours avec le numéro de version Windows NT de base. Copiez les fichiers suivants à partir du CD-ROM d'installation pour la version appropriée :

\SUPPORT\DEBUG\I386 C:\MYSYMBOLS [CD Drive] /S XCOPY.
3. Copiez des symboles pour les fichiers binaires appropriés de Service Pack au-dessus de votre arborescence personnalisée. Symboles de Service Pack figurent sur les serveurs répertoriés dans la section sous "Emplacements intitulés Symbol"
4. Copiez tous les correctifs tiers tels que les signes Compaq SSD au-dessus de dans votre arborescence de symboles personnalisée. Vous pouvez doit obtenir ces symboles à partir du site Web de fournisseur. REMARQUE : étapes 3 et 4 peut doit être inversé selon l'ordre qu'ils ont été installés par le client. Correspondez aux étapes du client.
5. Si un correctif est installé sur le serveur, vous devez vous procurer le symbole correspondant pour le correctif. Si le correctif n'est pas fourni à un fichier de symboles, vous devez contacter services de support technique Microsoft au sujet des possibilités de ces symboles. Copiez le symbole mis à jour au-dessus de votre arborescence de symboles personnalisée. Vérifiez que vous le placez dans le sous-dossier correct (par exemple Sys exe, dll, etc).

Utiliser des symboles dans le débogueur

Un Windows NT Debugger tel qu'I386KD.EXE recherche des symboles aux emplacements suivants :
Variable _NT_ALT_SYMBOL_PATH d'environnement système
Variable _NT_SYMBOL_PATH d'environnement système

Ces emplacements sont définis en passant par variables d'environnement système. Ils sont généralement configurés par un fichier de commandes de débogage utilisant la commande SET. Le _NT_ALT_SYMBOL_PATH est facultatif. Par exemple
Vous définissez _NT_SYMBOL_PATH = K:\NT35- SP3\SYMBOLS
Vous _NT_ALT_SYMBOL_PATH définissez = c:\WINNT\SYMBOLS

REMARQUE : symboles sont le répertoire au-dessus des sous-répertoire d'extension (si le fichier de noyau NTOSKRNL.DBG de symboles se trouve autrement dit à C:\DEBUG\511\I386\SYMBOLS\EXE\NTOSKRNL.DBG) Le _NT_SYMBOL_PATH a dû être défini sur C:\DEBUG\511\I386\SYMBOLS

Comment à des chemin d'accès Utiliser :

Ces chemin d'accès différents sont utilisés bon une utilisation pour conserver des arborescence de symboles statiques pour chaque version Windows NT. Vous pointez simplement vos chemin d'accès à symboles vers chaque version et Service Pack si nécessaire. Vous pouvez par exemple utiliser le suivant pour un système 1057 avec SP2 installé :
Vous définissez _NT_ALT_SYMBOL_PATH = c:\NT351- SP2\SYMBOLS
Vous définissez _NT_SYMBOL_PATH = K:\NT351- 1057\SYMBOLS

Le débogueur tente d'utiliser d'abord les symboles de Service Pack. [Les symboles de Service Pack n'incluent pas les symboles de génération de base dans ce cas seulement les symboles SP]. Si le débogueur ne trouve pas un symbole particulier dans l'arborescence SP, il le recherche dans l'arborescence de symboles 1057.

Ordre de recherche :

Les chemin d'accès différents à symboles sont analysés dans l'ordre indiqué au-dessus de. Le premier fichier de symboles avec le nom correct qui est trouvé est utilisé. Dans l'exemple au-dessus de _NT_ALT_SYMBOL_PATH = c:\NT351- SP2\SYMBOLS est une le premier recherché.

Override variable d'environnement :

Vous pouvez où spécifier un chemin d'accès à symboles, I386KD prend en charge un commutateur de ligne de commande "- y ". Toutefois utiliser ce commutateur ignore vos variables d'environnement existantes.

Modifiez le chemin d'accès à la volée :

Vous pouvez modifier le chemin de recherche de symbole à tout moment dans le débogueur en émettant " ! Sympath " commande. Par exemple
! SYMPATH C:\SYMBOLS.

Vérifier les symboles

! RECHARGEMENT :

" KD &gt; ! RECHARGEMENT " the "vous devez vous prompt pour la première fois taper" à une fois que vous avez obtenu. Cela provoque l'information symbolique est rechargée et est synchronisée. Si vous obtenez l'erreur "PsLoadedModuleList être NULL" vous comportez probablement les symboles incorrects chargé. Assurez-vous que vous avez installé le fichier HAL.DBG et NTOSKRNL.DBG approprié pour l'ordinateur que vous déboguez.

! PROCESSUS :

Des symboles peuvent aussi être vérifiés si vous obtenez l'erreur, vous comportez probablement les symboles incorrects "Impossible à trouver une tête de liste de processus" chargés " en tapant PROCESSUS " !.

KB :

Si le Stack Trace possède des intervalles dans la liste de noms de fonctions, cela indique que vous manquez des symboles. Que fonctions sont manquantes peut également indiquer une pile endommagée, mais cela est quelque peu rare. Vérifiez double vos symboles avant de déclarer une pile endommagée. Également regardez sur la pile aux adresses de retour. Ils tout doivent être supérieurs que 8000000.

! SYMPATH :

Des symboles incorrects peuvent être corrigés en accédant vers une autre fenêtre de commande, en copiant les symboles appropriés et en effectuant ! ! symbol path SYMPATH rechargement sur le débogueur ou par spécifier le chemin d'accès correct a.

Vérification de symbole avancé

Clients ne savent pas parfois quelles versions de pilote ils ont installées sur leurs ordinateurs. Ils ont peut-être installé un correctif pour NTFS, mais le numéro de bogue n'ils connaissent pas. Vous devez déterminer peut-être manuellement les fichiers .DBG sont nécessaires.

La façon cible et dbg meilleur vérifier absolument correspondance doit afficher leur valeur de somme de vérification. Cette valeur est stockée à l'intérieur du en-tête de fichier.

Remarque : pour plus savoir sur le système d'exploitation, format de fichier rechercher "format de fichier Portable Exécution" dans MSDN. L'en-tête PE contient des numéros de version vous liez date\time, etc.

D'abord déterminer le somme de vérification des fichiers clients et rechercher un fichier dbg doté d'un somme correspondant de vérification, puis fait le plan général ici.

Il existe nombreuses manières d'extraire le chksum de la cible et le dbg. Si vous avez l'accès aux fichiers, vous pouvez utiliser le suivant. Cet exemple suppose que vous êtes intéressé dans le fichier NTOSKRNL.

LINK32 :

Cet utilitaire se trouve dans le sous-répertoire NT Build, \MSTOOLS :
NTOSKRNL.DBG LINK32.EXE D'EN-TÊTES--LINK32.EXE NTOSKRNL.EXE D'EN-TÊTES- VIDAGE DE VIDAGE-

DUMPBIN :

Cet utilitaire se trouve dans le répertoire Visual C bin :
NTOSKRNL.DBG /HEADER /HEADER DUMPBIN.EXE NTOSKRNL.EXE DUMPBIN.EXE

Si vous utilisez un débogueur sur un système ou un vidage sur l'incident, vous trouvez le checksum du fichier cible en affichant l'en-tête de fichier dans la mémoire.
1. ! PILOTER donnera à vous l'adresse de base de pilote.
2. Baseaddr dd + L1 d8

Versions contrôlées

Si vous installez des versions debug vérifié de fichiers de Windows NT, vous devez effectuer dispositions spéciales pour ils des fichiers dbg de symboles. Certain correctif a vérifié que générations disposent de l'information symbolique être intégré au fichier cible. Dans ces cas simplement effectuez une copie du fichier et il nommez *.DBG. Placez le fichier de sous-répertoire de symbole normal autrement dit \SYMBOLS\EXE.

Toutes les générations de vérification publiée publiquement possèdent des fichiers dbg distinct uniquement similaires aux versions gratuites. Le dbg pour versions contrôlé et gratuits est cependant différent. Par exemple la MS NT DDK contient une version vérifiée complète de NT. Il contient également un symbole complet défini passant avec il.

REMARQUE : a vérifié que versions du noyau NT est même pour les systèmes Multiprocesseur et Monoprocesseur.
